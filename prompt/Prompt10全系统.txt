===========================================
A. 代码层面：全链路与系统衔接
===========================================

一、全链路端到端追踪（逐行跟代码，不要凭印象）

1. 行情数据链路：
   从 gateway/okx/ 的 WebSocket 收到一条行情消息开始，逐步追踪：
   解析 → 转为 core/models.py 中的数据对象 → event_bus 发布 → 
   bar_generator 合成 K 线 → strategy_engine 分发 → 策略的 on_bar 收到
   - 每个环节的数据类型是否一致？不要出现上游传 dict 下游期望对象的情况。
   - 是否有任何环节可能静默丢数据？

2. 订单执行链路：
   从策略调用 buy/sell 开始，逐步追踪：
   策略 → strategy_engine → risk_engine → gateway 发单 → 交易所返回 → 
   订单状态更新 → 仓位更新 → PnL 计算 → 回调策略 on_order/on_trade
   - 风控是否真的不可绕过？有没有任何代码路径跳过 risk_engine？
   - 订单状态更新和仓位更新的顺序是否正确？

3. 回测 vs 实盘路径对比：
   同一个策略代码，在回测和实盘中走的调用路径分别是什么？
   策略层应该完全一致，只有数据源和执行层不同。
   如果策略代码中有 if backtest/else live 的分支，这是架构问题，请消除。

二、状态一致性

4. 仓位状态：
   系统中有几个地方维护了仓位信息？（strategy_engine、risk_engine、gateway 可能各有一份）
   这些仓位的更新时机是否一致？哪一份是权威来源？其他份是否从权威来源同步？
   如果存在不一致风险，请修复——统一为单一仓位来源，其他地方引用它。

5. 资金状态：
   下单后是否冻结资金？撤单后是否释放？部分成交时冻结金额是否正确调整？

6. 断连后订单状态同步：
   网络断连导致成交回报丢失时，本地订单状态会卡住。
   重连后是否有机制查询并同步未知状态的订单？如果没有，请实现。

三、并发安全

7. 多策略并发下单：
   两个策略同时发出下单请求，risk_engine 的检查是否原子性的？
   比如各买 0.5 BTC，单独看都没超限，但合计超了。
   如果没有并发保护（锁或队列串行化），请添加。

8. 行情回调与订单回调交叉：
   策略的 on_bar 执行中，如果 on_order 回调同时到达，会发生什么？
   确认当前并发模型（单线程事件循环 or 多线程），评估是否安全。

9. WebSocket 消息堆积：
   策略处理慢时，WebSocket 消息是否会堆积？是否有缓冲机制？

四、异常恢复

10. 进程崩溃恢复：
    持有仓位时进程崩溃重启，如何重建仓位状态？
    至少应该在重启时从交易所 API 查询当前持仓。如果没有这个逻辑，请实现。

11. 交易所维护：
    OKX 维护期间 API 全部不可用，系统是否会疯狂重试？
    重试应有上限和退避，维护期间策略产生的信号应丢弃而非排队。

12. 网络抖动：
    WebSocket 短暂断连后重连，丢失的 K 线数据是否会补全？
    长时间断连是否自动暂停策略？

B. 交易操作层面：用真实场景验证逻辑
===========================================

以下每个场景，请在代码中追踪完整执行路径，验证结果是否正确。
如果发现问题，直接修复。

五、下单与成交

13. 开多仓场景：
    策略信号要开多 BTC-USDT 永续合约 0.1 BTC，当前价格 60000 USDT。
    追踪完整流程验证：
    - 风控检查：金额计算是否正确？（0.1 * 60000 = 6000 USDT）
    - 如果是合约交易，保证金计算是否正确？杠杆倍数是否被正确应用？
    - 下单参数组装：发给 OKX API 的参数（instId、tdMode、side、ordType、sz、px）是否正确？
    - 单位问题：OKX 合约的 sz 是张数还是币数？代码中是否正确处理了这个转换？
      （OKX 永续合约：BTC 合约面值 0.01 BTC/张，0.1 BTC = 10 张）

14. 平仓场景：
    持有多仓 0.1 BTC，策略信号平仓。
    - 平仓方向是否正确？（多仓平仓 = 卖出平仓，side=sell, posSide=long）
    - 平仓数量是否等于当前持仓量？不能多平也不能少平。
    - 如果用的是"双向持仓"模式（OKX 的 long/short mode），下单参数的 posSide 是否正确设置？
    - 如果用的是"单向持仓"模式（net mode），是否不需要 posSide？

15. 部分成交场景：
    下单买入 0.1 BTC，只成交了 0.06 BTC，剩余 0.04 BTC 未成交。
    - 仓位是否更新为 0.06 而非 0.1？
    - 剩余订单状态是否为 PartialFilled？
    - 如果后续 0.04 BTC 也成交了，仓位是否正确更新为 0.1？
    - 如果用户撤销剩余部分，冻结资金是否正确释放？

16. 止损/止盈订单：
    如果系统支持止损止盈，检查：
    - 止损价格的方向是否正确？（多仓止损应该 < 开仓价，空仓止损应该 > 开仓价）
    - 如果系统不支持止损止盈功能，跳过此项。

六、仓位与 PnL 计算

17. 多仓 PnL 计算：
    开多 0.1 BTC @ 60000，当前价 62000。
    - 浮动盈亏 = 0.1 * (62000 - 60000) = 200 USDT。代码计算结果是否正确？
    - 如果是合约，PnL 计算是否考虑了合约面值和杠杆？
    - 手续费是否从 PnL 中扣除？

18. 空仓 PnL 计算：
    开空 0.1 BTC @ 60000，当前价 58000。
    - 浮动盈亏 = 0.1 * (60000 - 58000) = 200 USDT。方向是否正确（空仓价格下跌是盈利）？
    - 最容易出 bug 的地方：空仓 PnL 的正负号。请仔细检查。

19. 多次加仓后的均价计算：
    第一次开多 0.1 BTC @ 60000，第二次加仓 0.05 BTC @ 61000。
    - 持仓均价应为 (0.1*60000 + 0.05*61000) / 0.15 = 60333.33 USDT。
    - 代码是否正确计算了加权平均价？

20. 多空翻转：
    当前持有多仓 0.1 BTC，策略信号开空 0.2 BTC。
    - 正确行为：先平多 0.1 BTC，再开空 0.1 BTC（或者一步到位净仓位 -0.1 BTC）。
    - 代码如何处理这种情况？是直接下一笔反向单还是分步处理？
    - 如果 OKX 设置的是双向持仓模式，这两笔是不同的操作（平多 + 开空），不能合并为一笔。

七、合约交易特有逻辑

21. 杠杆设置：
    下单前是否检查/设置了杠杆倍数？OKX 的杠杆是按品种设置的，
    代码是否在首次交易某个品种时调用 set-leverage API？

22. 保证金模式：
    代码中设置的是逐仓（isolated）还是全仓（cross）？
    这会影响保证金计算和强平价格。是否与 config 中的配置一致？

23. 强平风险：
    如果账户可用保证金不足，交易所会强平。系统是否监控了保证金率？
    至少应该在保证金率低于某个阈值时发出告警。如果没有，在 risk_engine 中添加。

24. 资金费率：
    永续合约每 8 小时有一次资金费率结算，这会影响账户余额和 PnL。
    系统是否考虑了资金费率的影响？
    - 回测中：是否模拟了资金费率的影响？如果没有，至少在回测报告中标注"未计入资金费率"。
    - 实盘中：资金费率结算后余额变动是否正确反映到系统中？

八、边界场景

25. 最小下单量：
    OKX 每个交易对有最小下单量限制（比如 BTC 永续最小 1 张 = 0.01 BTC）。
    如果策略计算出的下单量小于最小限制，系统如何处理？
    应该在下单前检查并拒绝，而非发到交易所被拒。

26. 价格精度：
    OKX 不同交易对的价格精度不同（BTC 是 0.1 USDT，有些山寨币是 0.0001 USDT）。
    下单价格是否按交易对的 tick size 做了精度处理（四舍五入到合法精度）？
    如果没有，从交易所拿到 instruments 信息后缓存，下单前校验和修正精度。

27. 数量精度：
    同上，下单数量也有精度限制。是否做了处理？

28. 余额不足：
    如果策略要求的下单金额超过了账户可用余额，系统是否在下单前检查？
    不能等交易所返回 insufficient balance 错误才处理。

九、Kill Switch 与安全兜底

29. Kill Switch 实战验证：
    模拟以下场景验证 emergency_stop() 是否正确工作：

    - 验证：2 笔挂单是否被撤销？0.1 BTC 是否被市价平仓？所有策略是否暂停？
    - 如果撤单 API 调用失败，是否重试而非放弃？

30. 配置防呆：
    - 实盘配置和回测配置是否有可能混用？
      （比如回测中手续费率设为 0，如果误用于实盘会导致 PnL 计算错误）
    - 实盘启动时是否会打印关键配置让用户确认？
      （至少应该打印：交易所、交易对、杠杆倍数、保证金模式、风控阈值）
