请审查 risk/ 目录，这是系统安全的最后防线：

1. risk/risk_engine.py：
   - 风控引擎是否串联在下单链路上？确认每一笔订单在发送到 gateway 之前都必须经过 risk_engine 检查。如果当前代码中存在绕过风控直接下单的路径，这是严重的架构缺陷，请修正。
   - 多个风控规则之间的执行顺序是否明确？建议按"成本从低到高"排列：rate_limiter（最快）→ order_validator → position_limit → loss_limit。如果当前顺序不合理请调整。
   - 在 risk_engine.py 中添加 emergency_stop() 方法（Kill Switch），功能为：
     a. 撤销所有未完成订单
     b. 市价平掉所有持仓
     c. 暂停所有策略
     d. 记录风控日志
     这是量化系统的必备安全能力，不是可选项。
   - 添加账户级最大回撤保护：当账户净值从历史最高点回撤超过配置的阈值时，自动触发 emergency_stop()。阈值从 settings.yaml 读取。

2. risk/order_validator.py：
   - 是否检查了价格偏离度？挂单价偏离最新市价超过 X% 则拒绝（防止 fat-finger 错误）。如果没有请添加，阈值可配置。
   - 是否检查了最小下单量？低于交易所最小下单量的订单提前拦截，避免发到交易所被拒。
   - 检查订单方向合理性：在已有多头持仓时是否允许再开空头（如果是对冲策略允许，否则应拦截）？当前逻辑是什么？确认是否合理。

3. risk/position_limit.py：
   - 是否区分了单策略持仓限制和账户总持仓限制？
   - 持仓限制是按币种还是按交易对计算？确认当前逻辑并在代码注释中说明设计决策。

4. risk/loss_limit.py：
   - 触发日亏损上限后的动作是什么？正确做法是：阻止新开仓但允许平仓。检查当前逻辑是否如此。
   - 亏损计算是否同时考虑已实现盈亏和浮动盈亏？建议两者都监控。
   - 日亏损的"日"是按 UTC 0 点重置还是 24 小时滚动窗口？确认并在代码中注释说明。

5. risk/rate_limiter.py：
   - 这个 rate_limiter 和 gateway 层可能存在的 API 限频是什么关系？
   - 如果 gateway 已经有 API 级别的限频，这里应该是业务层面的限频（比如同一策略不能在 1 秒内连续下 5 单），不应该重复实现 API 限频。明确两者职责边界。

6. 所有风控事件都要记录日志，包括：触发时间、触发的规则名、触发原因、当时的仓位/订单信息、采取的动作（拒绝/平仓/暂停）。

不新增文件，所有改动在现有文件内完成。
