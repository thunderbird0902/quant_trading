1. 本地数据存储（优先用 SQLite，不要引入 PostgreSQL/InfluxDB 等外部依赖）：
   
   修改 data/database.py，实现基于 SQLite 的本地 K 线数据存储，至少包含：
   - 建表：存储 K 线数据（交易对、周期、时间戳、open、high、low、close、volume），
     用 (symbol, interval, timestamp) 做联合主键，天然防重复。
   - write_bars(symbol, interval, bars)：批量写入 K 线数据。
   - read_bars(symbol, interval, start_ts, end_ts)：按时间范围查询，返回列表。
   - get_bar_range(symbol, interval)：查询本地已有数据的起止时间戳，
     用于判断哪些数据需要从 API 补拉。
   
   SQLite 文件放在项目根目录的 data/ 或 output/ 下，路径从 settings.yaml 读取。
   不要用 ORM，直接用 sqlite3 标准库即可。

2. 历史数据管理：

   修改 data/data_feed.py，添加一个方法 load_history(symbol, interval, start, end)，
   逻辑为：
   a. 先调 database.get_bar_range() 查本地已有范围
   b. 计算缺失的时间区间
   c. 只对缺失部分调用 gateway 的 REST API 拉取（注意 OKX 单次最多返回 300 根 K 线，
      需要分页循环拉取）
   d. 拉取结果写入本地 SQLite
   e. 最终从本地 SQLite 一次性读取完整数据返回
   
   这样回测时第一次会慢（需要拉取），之后就是纯本地读取，几乎秒级完成。

   - 写入失败不能导致主流程崩溃，用 try-except 包裹，错误记日志。
